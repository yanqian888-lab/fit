# 减肥搭子 APP —— 技术架构文档

## 1. 技术栈选型

### 1.1 选型原则

- 极致低成本：优先免费、开源、轻量方案
- 快速落地：减少学习成本，社区生态成熟
- 一套代码多端运行：降低维护成本

### 1.2 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端框架 | UniApp + Vue3 | 一套代码编译 H5、微信小程序、iOS、Android |
| UI 组件库 | uView UI 2.0 | 兼容 UniApp 的多端组件库 |
| 后端框架 | Node.js + Express | 轻量、易部署、社区活跃 |
| 本地数据库 | SQLite | 零配置、本地存储、免费 |
| 云端数据库 | MySQL 5.7+ | 云端备份、可选部署 |
| ORM | Sequelize / better-sqlite3 | 统一操作 SQLite 和 MySQL |
| AI 服务 | 豆包开放平台 API | 优先使用免费模型 |
| 文件存储 | 腾讯云 COS | 50GB 永久免费额度 |
| 部署 | 轻量应用服务器 | 2 核 2G、Ubuntu 22.04 |
| 进程管理 | PM2 | Node 进程守护 |
| 反向代理 | Nginx | 静态资源、HTTPS、负载均衡 |

## 4. 关键设计决策

### 4.1 数据存储策略

| 数据类型 | 存储位置 | 原因 |
|----------|----------|------|
| 用户资料、聊天记录、沉淀记录 | SQLite（本地） | 零成本、读写快、无需运维 |
| 云端备份 | MySQL（可选） | 多设备同步、数据安全 |
| 图片、文件 | 腾讯云 COS | 大文件不适合存数据库 |
| 配置、Token | 服务器内存 / Redis 可选 | MVP 阶段用内存即可 |

### 4.2 AI 调用策略

- 所有 AI 调用通过后端统一转发，前端不直接暴露 API Key
- 主协调 Agent 优先快速响应，沉淀和计算异步处理
- 使用提示词工程 + JSON Schema 约束信息沉淀输出
- 对常见计算（BMI、BMR、热量缺口）完全本地计算，不调用大模型

### 4.3 成本控制策略

| 项目 | 成本策略 |
|------|----------|
| AI 调用 | 优先豆包免费模型，控制上下文长度 |
| 服务器 | 2 核 2G 轻量服务器 |
| 存储 | SQLite 本地 + COS 免费额度 |
| 推送 | 不使用第三方推送，用小程序订阅消息（后续） |
| 短信 | MVP 阶段不做手机号登录 |

### 4.4 多端适配策略

- UniApp 条件编译处理平台差异
- 微信小程序优先，H5 用于测试和分享
- iOS / Android 在 MVP 后打包上架

## 5. 安全设计

### 5.1 认证授权

- MVP 阶段使用微信小程序 `wx.login` 获取 `openid`
- 后端生成 JWT Token 返回给前端
- 后续支持手机号登录时引入短信验证

### 5.2 数据安全

- 敏感配置放 `.env`，不提交代码
- 用户密码（如有）使用 bcrypt 加密
- 数据库文件定期备份到 COS
- API 接口做基础限流（如每分钟 60 次）

### 5.3 内容安全

- 用户图片上传前做类型和大小校验
- 聊天内容过滤违法违规信息（可接入微信内容安全接口）

## 6. 性能优化

### 6.1 前端

- 聊天记录分页加载
- 图片懒加载和压缩
- 静态资源 CDN 加速

### 6.2 后端

- AI 调用异步处理，不阻塞主线程
- 使用连接池管理数据库连接
- Nginx 静态缓存

### 6.3 AI

- 控制提示词长度，避免长上下文
- 对常见查询使用本地缓存
- 合理设置 max_tokens

## 8. 技术风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 豆包免费额度用完 | 成本上升 | 监控调用量，设置告警，必要时降级 |
| 微信小程序审核不通过 | 无法上线 | 提前阅读审核规范，准备隐私政策 |
| 自动沉淀准确率低 | 用户体验差 | 持续优化提示词，增加人工确认机制 |
| 服务器性能不足 | 响应慢 | 先垂直升级，后续再考虑分布式 |
| iOS 上架周期长 | 延迟发布 | 先发布微信小程序验证市场 |
